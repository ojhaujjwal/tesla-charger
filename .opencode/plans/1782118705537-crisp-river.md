# Plan — Deterministic CI auto-gate plugin for opencode

## Goal

Deterministically run `npm run ci` whenever opencode edits files in a session,
unless opencode itself already ran `npm run ci` in that same turn; on failure,
feed the CI output back to opencode as a new user turn so it can react,
capped at 2 consecutive auto-fix turns per session (reset on a real user
prompt) to prevent runaway loops.

Determinism comes from a TypeScript plugin hook that runs **outside** the LLM
prompt path — no prompt-instruction gymnastics. The model can't forget or skip
it. This is the only mechanism in opencode that gives guaranteed, scripted
side effects on tool/turn boundaries.

## Confirmed facts (from opencode source)

  - `session.next.shell.started` — properties `{ command, callID, messageID }`

  - `session.idle` — properties `{ sessionID }` (`types.gen.ts:475-480`), fired

    `processor.ts:931,957` and `run-state.ts:62,82`). The deterministic
    "assistant turn just ended" boundary.
- `client.prompt({ path: { id }, body: { parts: [{ type: "text", text }] } })`
  injects a new user turn that re-engages the assistant

- `package.json` already defines `npm run ci`
  (`typecheck && lint && test -- --run && format-fix`). `.opencode/package.json`
  already pulls `@opencode-ai/plugin@1.14.29`, so `import type { Plugin } from
  "@opencode-ai/plugin"` works out of the box.

## Decisions (confirmed with user)

| Decision | Choice |
|---|---|
| Feedback on failure | Inject new assistant turn via `client.prompt` |
| "Already ran tests" suppression | `npm (run )?ci` only (exact `ci` script) |
| Loop cap | 2 consecutive auto-fix turns per session; reset on real user prompt |
| Edit detection | Only `file.edited` (edit / write / apply_patch), not shell-driven edits |

## Implementation

### File: `.opencode/plugin/ci-gate.ts` (new, tracked in git)

Auto-discovered, no `opencode.json` change needed.

```ts
import type { Plugin } from "@opencode-ai/plugin"

const CI_TIMEOUT_MS = 5 * 60_000
const MAX_AUTOFIX_TURNS = 2
const SENTINEL = "[CI-GATE-AUTO]" // prefix we stamp on injected prompts

type TurnState = { edited: boolean; ciRan: boolean }
const turn = new Map<string, TurnState>()        // per session, current un-processed turn
const autofixCount = new Map<string, number>()   // per session, consecutive auto-fixes
const inflightAutoRun = new Set<string>()        // sid guard: don't recurse on our own idle

const isCiCommand = (cmd: string) =>
  /^npm\s+(?:run\s+)?ci(?:\s|$)/.test(cmd.trim())

const RESET = "\x1b[0m"; const RED = "\x1b[31m"; const GREEN = "\x1b[32m"; const DIM = "\x1b[2m"

export default (async ({ client, $, directory }) => {
  const runCmd = $`npm run ci`.cwd(directory)     // Bun shell, pinned to project root

  return {
    event: async ({ event }) => {
      const ev = event as { type: string; properties: any }

      // 1) track edits + the assistant's own CI runs during this turn
      if (ev.type === "file.edited") {
        const sid = ev.properties.sessionID
        const s = turn.get(sid) ?? { edited: false, ciRan: false }
        s.edited = true
        turn.set(sid, s)
        return
      }
      if (ev.type === "session.next.shell.started") {
        const sid = ev.properties.sessionID
        const s = turn.get(sid) ?? { edited: false, ciRan: false }
        if (isCiCommand(ev.properties.command ?? "")) s.ciRan = true
        turn.set(sid, s)
        return
      }

      // 2) at turn-end, run CI ourselves if needed
      if (ev.type !== "session.idle") return
      const sid = ev.properties.sessionID
      const st = turn.get(sid) ?? { edited: false, ciRan: false }
      turn.delete(sid)  // reset per-turn trackers for the next pass

      if (!st.edited || st.ciRan) return          // no work this turn
      if (inflightAutoRun.has(sid)) return        // guard against recursion
      const count = autofixCount.get(sid) ?? 0
      if (count >= MAX_AUTOFIX_TURNS) {
        console.error(`${RED}[ci-gate]${RESET} cap reached (${count}); not auto-running. Run ${DIM}npm run ci${RESET} manually.`)
        return
      }

      inflightAutoRun.add(sid)
      try {
        let stdout = "", exitCode = 0, ok = false
        try {
          stdout = await runCmd.text()
          ok = true
        } catch (e: any) {
          stdout = (e?.stdout ?? "") + (e?.stderr ?? "") + (e?.message ?? String(e))
          exitCode = e?.exitCode ?? 1
        }

        if (ok) {
          console.log(`${GREEN}[ci-gate]${RESET} npm run ci PASSED (auto).`)
          autofixCount.delete(sid)   // a pass clears the streak
          return
        }
        autofixCount.set(sid, count + 1)
        const tail = stdout.length > 12_000 ? "…" + stdout.slice(-12_000) : stdout
        const body = `${SENTINEL}\n` +
          `Automated CI gate ran \`npm run ci\` after your edits this turn. ` +
          `It failed with exit code ${exitCode}. Fix all failures listed below ` +
          `(typecheck/lint/test/format), then either re-run \`npm run ci\` yourself ` +
          `or stop and let the gate run again.\n\n<ci-output>\n${tail}\n</ci-output>\n`
        await client.prompt({
          path: { id: sid },
          body: { parts: [{ type: "text", text: body }] },
        })
      } finally {
        inflightAutoRun.delete(sid)
      }
    },

    // 3) reset the autofix streak when a real (non-injected) user prompt arrives
    "chat.message": async (input: any, _output: any) => {
      const sid: string | undefined = input?.sessionID
      if (!sid) return
      const text: string = input?.text ?? input?.parts?.[0]?.text ?? ""
      if (text.startsWith(SENTINEL)) return    // ours; don't reset
      autofixCount.delete(sid)
    },
  }
}) satisfies Plugin
```

### Behavioural contract

1. Per turn (sessionID-scoped):
   - `file.edited` → mark `edited = true`.
   - `session.next.shell.started` with command matching `npm (run )?ci` → mark
     `ciRan = true`.
   - `session.idle`:
     - If `!edited || ciRan` → do nothing.
     - Else run `npm run ci` synchronously, in project root, with ~5 min
       timeout (Bun's `$` supports `.timeout()` — wire it in).
     - Pass → log `[ci-gate] PASSED`, clear that session's streak counter.
     - Fail → bump the streak counter; if still under the cap, `client.prompt`
       the failure output back into the session (sentinel-prefixed so step 3
       doesn't reset). If at/over the cap, log to stderr and stop.
2. Streak count resets to 0 on either a passing auto-CI run or a real user
   message (detected by the absence of the `SENTINEL` prefix on the incoming
   `chat.message`).
3. `inflightAutoRun` belt-and-suspenders guard prevents the plugin's own
   `client.prompt` call from re-entering the idle handler in the same instant.

### Why `event` instead of `tool.execute.after`

- `tool.execute.after` for `edit` would inject per-edit and run `npm run ci`
  once per file — too slow. `event` + `session.idle` runs once per turn after
  the model is done, before the user types. Matches the user's intent.
- `file.edited` covers `edit`, `write`, and `apply_patch` simultaneously and
  is the documented stable surface. We avoid parsing `edit`'s `args.filePath`
  shape.

## Verification end-to-end

After saving the plugin, **quit and restart opencode** (config is loaded once
at startup; the running session won't pick up new plugins).

1. **Happy path: model edits + doesn't run CI.**
   - Start a fresh chat in `/Users/uo/Projects/tesla-charger`.
   - Ask opencode to make a benign edit, e.g. "Add a one-line comment to
     `src/tests/some-test.ts`" intentionally NOT telling it to run tests.
   - Expected: on turn end, terminal prints `[ci-gate] npm run ci PASSED
     (auto).` The user never had to type `npm run ci`.
2. **Suppression: model runs CI itself.**
   - Ask opencode: "Add a comment to `src/...`, then run `npm run ci` and
     report the result."
   - Expected: `session.next.shell.started` sets `ciRan = true`; the gate
     detects this and does **not** run CI again. Log line absent.
3. **Failure feedback loop (cap = 2).**
   - Inject a deliberate typecheck error via a prompt that does not run CI,
     e.g. "Add `const broken: NotAType = 1` to an existing source file."
   - Expected: gate runs CI, fails, and re-prompts the model with the
     failure; model fixes it; if next run still fails, gate injects one more
     time (streak = 2); the third idle with still-broken state logs
     `[ci-gate] cap reached` to stderr instead of injecting.
4. **Cap reset on real user prompt.**
   - After step 3 logs "cap reached", send any manual prompt. Expected:
     `autofixCount` resets; subsequent failing turns get fresh retries.
5. **Pass-recovery clears the streak.**
   - At count = 1, make an edit that introduces a transient failure, then
     immediately fix it. On the next auto-CI pass, the streak clears. Verify
     by re-introducing a failure on the following turn — gate retries from 0.
6. **Sanity: no false positives from shell edits.** Ask opencode to run
   `git stash` or `echo > /tmp/x` without editing tracked source — no auto-CI
   runs (because `file.edited` never fired).

## Known limitations / agreed tradeoffs

- The gate runs `npm run ci` once per assistant turn that edited files. CI
  on this project takes ~30s-1m; turn-end wait is acceptable per the user.
- Detecting "opencode already ran CI" is anchored to the script name `ci`.
  If the assistant runs `npm test` or `npm run typecheck` instead, the gate
  will still run full `npm run ci` per the user's explicit choice ("Only
  `npm run ci`").
- Shell-driven file changes (`sed -i`, `git checkout -- src/x`) are
  intentionally not counted, per user's choice. The gate trusts the audit
  trail of `edit`/`write`/`apply_patch` tools as the source of truth.
- If a future opencode version renames the shell tool or event shapes, this
  plugin will need a one-line update. All hook shapes referenced here are
  stable as of the checked-out `packages/core/src/session/event.ts`.
