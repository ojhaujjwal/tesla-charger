# Plan — Deterministic CI auto-gate plugin for opencode

## Status: implemented and verified working

The plugin already exists at `.opencode/plugin/ci-gate.ts` and has been tested end-to-end. This document records the final design after iterating through three detection approaches.

## Goal

Deterministically run `npm run ci` whenever opencode edits files in a session, unless opencode itself already ran `npm run ci` in that same turn; on failure, feed the CI output back to opencode as a new user turn so it can react, capped at 2 consecutive auto-fix turns per session (reset on a real user prompt) to prevent runaway loops.

Determinism comes from a TypeScript plugin hook that runs **outside** the LLM prompt path — no prompt-instruction gymnastics. The model can't forget or skip it.

## Detection method: B (git stash create --include-untracked content hash)

Snapshot the working-tree content SHA at turn start (`chat.message` hook) and compare at turn end (`session.idle` event). Any content change — same file edited, new file added, file reverted — produces a different SHA. Net-zero edits (assistant reverted) correctly skip.

This was chosen over alternatives after iterating:

- **Approach tried first: `tool.execute.after` for `edit`/`write`/`apply_patch`.** Failed in real testing — when asked to edit without reading first, opencode used `bash`+`sed` instead, bypassing the gate entirely. DB inspection of historic tool calls confirmed this is routine: ~8% of file edits in this project's history go through bash (sed/python/tee/git checkout), not the edit tools.
- **Approach tried second: `tool.execute.after` + enumerated bash mutation patterns.** Too brittle. 1232 historic bash mutation commands across many patterns (sed/awk/perl/python/tee/cat>/>>/cp/mv/rm/git checkout/git restore/git stash/heredoc/dd/...). Each new creative method requires a plugin edit.
- **Approach tried third: `git status --porcelain` diff.** User pointed out: only compares file lists/status flags, not content. If `M src/foo.ts` is already dirty at turn start, an assistant edit to `src/foo.ts` produces the same `M src/foo.ts` line → gate sees no change → skips. **Bug.**
- **Final approach: `git stash create --include-untracked` SHA.** Content-hashes the full working tree including untracked files. Verified:
  - Clean tree → empty string
  - Dirty tracked file → SHA
  - Untracked file → SHA (with `--include-untracked`)
  - Two content-identical states → same SHA (correct no-op skip)

## Confirmed facts (from opencode source)

- Plugin contract: `packages/plugin/src/index.ts:222-335` (the `Hooks` interface). Auto-discovered loaders: any `*.ts`/`*.js` under `.opencode/plugin/` or `.opencode/plugins/` is loaded without any config entry (`config/plugin.ts:18-29` scans `{plugin,plugins}/*.{ts,js}`).
- `PluginInput` (`index.ts:56-66`) provides `client` (HTTP SDK client) and `directory` (project root).
- `event` hook receives every bus event (`plugin/index.ts:251-258`).
- `chat.message` hook (`plugin/src/index.ts:234-243`) fires for every prompt received — usable to distinguish a real user message from one we injected (we tag ours with a sentinel prefix).
- `client.session.prompt({ path: { id }, body: { parts: [{ type: "text", text }] } })` injects a new user turn that re-engages the assistant.
- `session.idle` (`sdk/js/src/gen/types.gen.ts:475-480`) fires when the session goes idle (assistant turn done, control returns to user).
- `session.next.shell.started` — properties `{ command, callID, messageID, sessionID }` (`packages/core/src/session/event.ts:152-162`). Best signal for detecting the assistant's bash command without parsing tool args.

## Decisions (confirmed with user)

| Decision | Choice |
|---|---|
| Detection method | B: `git stash create --include-untracked` content hash |
| Feedback on failure | Inject new assistant turn via `client.session.prompt` |
| "Already ran tests" suppression | `npm (run )?ci` only (detected via `session.next.shell.started` event) |
| Loop cap | 2 consecutive auto-fix turns per session; reset on real user prompt or passing auto-CI |
| CI invocation | **Synchronous** via `spawnSync` (not async Bun shell) — async `await` was cancelled by quick instance disposal in `opencode run` headless mode |

## Critical implementation detail: synchronous CI invocation

The first working version used `$\`npm run ci\`...await` (Bun shell). In testing, the hook fired but the async CI run was cancelled before completing because opencode disposes the instance ~35ms after `session.idle` (visible in logs as the `disposing instance` line immediately after `session.idle`). The hook dispatcher uses `void hook["event"]?.(...)` (fire-and-forget), so pending promises die with the instance.

**Fix:** use `spawnSync("npm", ["run", "ci"], { timeout: 5*60_000 })` to block the event loop until CI completes. The plugin's `event` hook is awaited sequentially by `Plugin.trigger`, so synchronous work in the hook delays the event-subscriber return until CI is done. This is the only way the gate runs to completion in `opencode run` mode.

In interactive / `opencode serve` mode, the instance persists across idle events, so the synchronous approach also works (just blocks the event loop briefly).

## File: `.opencode/plugin/ci-gate.ts` (final version)

```ts
import type { Plugin } from "@opencode-ai/plugin"
import { execSync, spawnSync } from "node:child_process"

const CI_TIMEOUT_MS = 5 * 60_000
const MAX_AUTOFIX_TURNS = 2
const SENTINEL = "[CI-GATE-AUTO]"

const RESET = "\x1b[0m"
const RED = "\x1b[31m"
const GREEN = "\x1b[32m"
const DIM = "\x1b[2m"

const lastSeenHash = new Map<string, string>()
const ciRanThisTurn = new Set<string>()
const autofixCount = new Map<string, number>()
const inflightAutoRun = new Set<string>()

const isCiCommand = (cmd: string) => /^npm\s+(?:run\s+)?ci(?:\s|$)/.test(cmd.trim())

const treeHash = (directory: string): string => {
  try {
    return execSync("git stash create --include-untracked", {
      cwd: directory,
      encoding: "utf8",
      timeout: 5_000,
      stdio: ["pipe", "pipe", "pipe"],
    }).trim()
  } catch {
    return ""
  }
}

const firstTextPart = (parts: any[]): string => {
  for (const p of parts ?? []) {
    if (p && typeof p === "object" && p.type === "text" && typeof p.text === "string") return p.text
  }
  return ""
}

const runCiSynchronously = (directory: string): { exitCode: number; stdout: string } => {
  try {
    const result = spawnSync("npm", ["run", "ci"], {
      cwd: directory,
      encoding: "utf8",
      timeout: CI_TIMEOUT_MS,
      stdio: ["pipe", "pipe", "pipe"],
      shell: process.platform === "win32",
    })
    return {
      exitCode: result.status ?? 1,
      stdout: (result.stdout ?? "") + (result.stderr ?? ""),
    }
  } catch (e: any) {
    return { exitCode: -1, stdout: `Failed to spawn npm run ci: ${e?.message ?? e}` }
  }
}

export default (async ({ client, directory }) => {
  return {
    "chat.message": async (input, output) => {
      const sid: string | undefined = input?.sessionID
      if (!sid) return
      const text = firstTextPart(output?.parts ?? []) || (output?.message as any)?.text || ""
      if (text.startsWith(SENTINEL)) return
      lastSeenHash.set(sid, treeHash(directory))
      autofixCount.delete(sid)
      ciRanThisTurn.delete(sid)
    },

    event: async ({ event }) => {
      const ev = event as { type: string; properties: any }

      if (ev.type === "session.next.shell.started") {
        const sid: string = ev.properties?.sessionID
        if (sid && isCiCommand(ev.properties?.command ?? "")) ciRanThisTurn.add(sid)
        return
      }

      if (ev.type !== "session.idle") return

      const sid: string = ev.properties?.sessionID
      if (!sid || inflightAutoRun.has(sid)) return

      const current = treeHash(directory)
      const last = lastSeenHash.get(sid) ?? ""
      const changed = current !== last

      if (ciRanThisTurn.has(sid)) {
        console.log(
          `${DIM}[ci-gate]${RESET} working tree changed but assistant already ran \`npm run ci\`; skipping.`,
        )
        lastSeenHash.set(sid, current)
        ciRanThisTurn.delete(sid)
        return
      }

      lastSeenHash.set(sid, current)
      ciRanThisTurn.delete(sid)

      if (!changed) return

      const count = autofixCount.get(sid) ?? 0
      if (count >= MAX_AUTOFIX_TURNS) {
        console.error(
          `${RED}[ci-gate]${RESET} autofix cap reached (${count}); not injecting. ` +
            `Run ${DIM}npm run ci${RESET} manually, or send any prompt to reset the cap.`,
        )
        return
      }

      inflightAutoRun.add(sid)
      try {
        console.log(`${DIM}[ci-gate]${RESET} running \`npm run ci\` synchronously (files changed this turn)...`)
        const { exitCode, stdout } = runCiSynchronously(directory)
        lastSeenHash.set(sid, treeHash(directory))

        if (exitCode === 0) {
          console.log(`${GREEN}[ci-gate]${RESET} \`npm run ci\` PASSED (auto) after detected file changes.`)
          autofixCount.delete(sid)
          return
        }

        const nextCount = count + 1
        autofixCount.set(sid, nextCount)
        console.error(
          `${RED}[ci-gate]${RESET} \`npm run ci\` FAILED (exit ${exitCode}); ` +
            `injecting feedback turn ${nextCount}/${MAX_AUTOFIX_TURNS}.`,
        )

        const tail = stdout.length > 12_000 ? "…" + stdout.slice(-12_000) : stdout
        const body =
          `${SENTINEL}\n` +
          `The CI gate plugin detected file changes this turn and auto-ran \`npm run ci\`, ` +
          `which failed with exit code ${exitCode}. Read the failure output below, fix every ` +
          `issue (typecheck / lint / test / format), then either re-run \`npm run ci\` ` +
          `yourself or stop editing so the gate can re-run. Do not reply with only text — ` +
          `make the code fixes.\n\n<ci-output>\n${tail}\n</ci-output>\n`

        await client.session.prompt({
          path: { id: sid },
          body: { parts: [{ type: "text", text: body }] },
        })
      } finally {
        inflightAutoRun.delete(sid)
      }
    },
  }
}) satisfies Plugin
```

### Behavioural contract

1. Per turn (sessionID-scoped):
   - `chat.message` → snapshot `git stash create --include-untracked` SHA, reset streak counter and `ciRanThisTurn`. (Skip if the message starts with `SENTINEL` — that's our own injected prompt.)
   - `session.next.shell.started` with command matching `npm (run )?ci` → mark `ciRanThisTurn`.
   - `session.idle`:
     - Compute current tree SHA. If unchanged from start → do nothing.
     - If `ciRanThisTurn` → log skip, advance snapshot, return.
     - Else run `npm run ci` **synchronously**. Pass → log, clear streak. Fail → bump streak; if still under cap, `client.session.prompt` the failure output back into the session. If at/over cap, log to stderr and stop.
2. Streak count resets to 0 on either a passing auto-CI run or a real user message (detected by absence of the `SENTINEL` prefix on the incoming `chat.message`).
3. `inflightAutoRun` guard prevents recursion on our own `client.session.prompt` call.

**Already verified end-to-end in testing:**

```
$ opencode run --print-logs "Add '// TEST' as first line of src/tests/unit/domain/timing.test.ts using sed. Don't read. Then stop."
[ci-gate] running `npm run ci` synchronously (files changed this turn)...
[ci-gate] `npm run ci` FAILED (exit 1); injecting feedback turn 1/2.
```

The gate detected the sed-driven edit (which would have bypassed Approach A), ran CI synchronously, and injected the failure back to the model. This confirms:
- Auto-discovery works (plugin loaded without any `opencode.json` change)
- Hooks fire correctly (`chat.message`, `event` for `session.next.shell.started` and `session.idle`)
- Synchronous CI invocation avoids the instance-disposal cancellation bug
- `client.session.prompt` successfully injects a feedback turn

**Verification steps to re-run manually:**

1. **Happy path: model edits + doesn't run CI.** Prompt: "Add a comment to `src/tests/...test.ts`, then stop." Expected: `[ci-gate] npm run ci PASSED (auto)`.
2. **Suppression: model runs CI itself.** Prompt: "Add a comment, then run `npm run ci` and report." Expected: `[ci-gate] working tree changed but assistant already ran \`npm run ci\`; skipping.`
3. **Failure feedback loop (cap = 2).** Prompt: "Add `const broken: NotAType = 1` to a source file." Expected: gate fails, re-prompts model; if next run still fails, gate injects one more time (streak = 2); the third failing turn logs `[ci-gate] cap reached` instead of injecting.
4. **Cap reset on real user prompt.** After step 3 logs "cap reached", send any manual prompt. Expected: `autofixCount` resets; subsequent failing turns get fresh retries.
5. **Pass-recovery clears streak.** At count = 1, fail transiently then fix. Next auto-CI pass clears the streak.
6. **Bash-driven sed edit (the slip-through case for Approach A).** Prompt: "Use bash sed to prepend `// TEST` as first line of a test file. Don't read first." Expected: gate fires (because tree content changed).

## Known limitations

- **False positive risk:** if something other than the assistant modifies the working tree during an active turn (user manually editing in another terminal, a git pull, a write-hook), the gate will fire. In this project: formatter disabled, LSP disabled, no write-hooks → risk is near-zero.
- **Suppressing auto-run when assistant already ran CI** is anchored to the exact script name `ci` (via `session.next.shell.started` event). If the assistant runs `npm test` or `npm run typecheck` instead, the gate will still run full `npm run ci` per the user's explicit choice ("Only `npm run ci`").
- **Synchronous CI blocks the event loop.** In interactive/serve mode, this means no other event processing during the CI run (~30s-1m on this project). Acceptable per the user; avoids the async-cancellation bug.
- **`git stash create` requires git.** If the project isn't a git repo, `treeHash` returns empty string always → gate never fires. Acceptable (this project is git-tracked).
